## 一、Python 与网络爬虫原理和策略

### 1.1 Python 核心基础

#### 数据类型与内存管理
- **可变类型 vs 不可变类型**：list/dict/set（可变） vs int/str/tuple（不可变）
- **深拷贝与浅拷贝**：`copy.copy()` vs `copy.deepcopy()`，引用传递的本质
- **GIL（全局解释器锁）**：CPython 中同一时刻只有一个线程执行 Python 字节码；CPU 密集型用多进程，IO 密集型用多线程/协程
- **垃圾回收**：引用计数为主，标记清除和分代回收为辅；循环引用问题及 `weakref` 的使用
- **装饰器与生成器**：`@decorator` 本质是闭包；`yield` 实现惰性求值，节省内存

#### 异步编程
- **asyncio 事件循环**：`async/await` 语法、`Task`、`Future`、`EventLoop`
- **协程 vs 线程 vs 进程**：协程切换无系统调用开销，适合高并发 IO 场景
- **aiohttp**：异步 HTTP 客户端，配合 asyncio 实现高并发爬虫
- **信号量控制并发**：`asyncio.Semaphore` 限制同时请求数

#### 多线程与多进程
- `threading` 模块：`Thread`、`Lock`、`RLock`、`Event`、`Condition`
- `multiprocessing` 模块：`Process`、`Pool`、`Queue`、`Pipe`
- `concurrent.futures`：`ThreadPoolExecutor`、`ProcessPoolExecutor`、`Future`
- **线程安全**：共享资源的锁保护、死锁的产生与预防

### 1.2 爬虫原理

#### HTTP 请求生命周期
```
DNS解析 → TCP三次握手 → (TLS握手) → 发送HTTP请求 → 服务器处理 → 返回HTTP响应 → 渲染/解析
```

#### 爬虫工作流程
```
URL管理 → 请求发送 → 响应获取 → 数据解析 → 数据存储
```

#### URL 调度器（URL Manager）
- **待爬取集合**：使用 `set` 或 Redis 的 `set` 去重
- **已爬取集合**：Bloom Filter 大规模去重（空间效率高，有误判率）
- **优先级队列**：`heapq` 实现 BFS/优先级爬取策略

### 1.3 爬虫核心策略

#### 抓取策略
| 策略 | 说明 | 适用场景 |
|------|------|----------|
| BFS（广度优先） | 逐层扩展 | 通用爬虫 |
| DFS（深度优先） | 一条路径到底 | 特定信息追踪 |
| Best-First | 按优先级排序 | 主题爬虫 |
| 增量爬取 | 只爬新内容 | 新闻/论坛监控 |

#### 反爬与反反爬
- **User-Agent 轮换**：维护 UA 池随机切换
- **IP 代理池**：轮询代理，避免单 IP 高频被封
- **Cookie 管理**：维护 Cookie 池，模拟登录态
- **请求频率控制**：随机延时（`time.sleep` + random），模拟人类行为
- **验证码识别**：OCR（Tesseract）、第三方打码平台、深度学习模型
- **JS 渲染页面**：Selenium / Playwright / Pyppeteer 无头浏览器
- **字体反爬**：识别自定义字体映射（如大众点评、猫眼）

#### Robots 协议
- `robots.txt` 是约定而非强制，`urllib.robotparser` 可解析
- 遵守 `Crawl-delay`、`Disallow` 等规则

### 1.5 Scrapy 框架详解

#### 架构组件
```
Engine（引擎）
  ├── Spider（爬虫）：定义爬取逻辑
  ├── Scheduler（调度器）：URL 去重与排序
  ├── Downloader（下载器）：发送请求获取响应
  ├── Item Pipeline（管道）：数据清洗与存储
  └── Middlewares（中间件）
       ├── Spider Middleware：Spider 与 Engine 之间
       └── Downloader Middleware：Downloader 与 Engine 之间
```

#### 关键概念
- **Request/Response 对象**：`meta` 传递数据、`callback` 指定回调
- **Item & ItemLoader**：数据容器与数据加载/清洗
- **去重机制**：基于 `dupefilter`（默认使用 SHA1 指纹）
- **Settings 优先级**：default < project < spider < cmd
- **CrawlSpider vs Spider**：`Rule` + `LinkExtractor` 自动提取链接
- **信号（Signals）**：`spider_opened`、`spider_closed`、`item_scraped` 等
- **Feed Export**：直接导出 JSON/CSV/XML

### 2.3 HTTP 协议

#### 请求方法
| 方法 | 说明 | 幂等 | 安全 |
|------|------|------|------|
| GET | 获取资源 | 是 | 是 |
| POST | 提交数据 | 否 | 否 |
| PUT | 替换资源 | 是 | 否 |
| DELETE | 删除资源 | 是 | 否 |
| HEAD | 获取响应头 | 是 | 是 |
| PATCH | 部分修改 | 否 | 否 |
| OPTIONS | 查询支持方法 | 是 | 是 |

#### HTTP 状态码
| 状态码 | 含义 | 爬虫关注点 |
|--------|------|-----------|
| 200 | OK | 正常响应 |
| 301 | 永久重定向 | 更新 URL |
| 302 | 临时重定向 | 跟踪 Location |
| 304 | 未修改（缓存） | 可用缓存 |
| 403 | 禁止访问 | 可能被反爬 |
| 404 | 未找到 | URL 失效 |
| 429 | 请求过多 | 需要降频 |
| 500 | 服务器错误 | 重试 |
| 503 | 服务不可用 | 等待重试 |

#### 关键 Header
- **请求头**：`User-Agent`、`Referer`、`Cookie`、`Accept`、`Accept-Language`、`Authorization`
- **响应头**：`Set-Cookie`、`Content-Type`、`Content-Encoding`（gzip/br）、`Cache-Control`
- **CORS 相关**：`Origin`、`Access-Control-Allow-Origin`

#### HTTP/1.1 vs HTTP/2 vs HTTP/3
- **HTTP/1.1**：持久连接（keep-alive）、管道化（有队头阻塞）
- **HTTP/2**：多路复用、头部压缩（HPACK）、服务器推送、二进制帧
- **HTTP/3**：基于 QUIC（UDP），解决 TCP 队头阻塞

#### Cookie 与 Session
- **Cookie**：存储在客户端，`Set-Cookie` 设置，随请求发送
- **Session**：服务端存储，通过 `session_id`（Cookie 中）关联
- **Token（JWT）**：无状态认证，`Authorization: Bearer <token>`

### 2.4 SSL/TLS

#### TLS 握手过程（TLS 1.2）
```
Client Hello (支持的加密套件、随机数)
    ↓
Server Hello (选定加密套件、随机数、证书)
    ↓
Client 验证证书 → 生成预主密钥 → 用服务器公钥加密发送
    ↓
双方用随机数+预主密钥生成会话密钥
    ↓
对称加密通信开始
```

#### TLS 1.3 改进
- 握手从 2-RTT 减少到 1-RTT（甚至 0-RTT 恢复）
- 移除不安全算法（RSA 密钥交换、3DES 等）
- 只保留 AEAD 加密（AES-GCM、ChaCha20-Poly1305）

#### 爬虫中的 SSL 问题
- **证书验证失败**：`verify=False`（不推荐）或指定 CA 证书
- **SSL 指纹检测**：JA3 指纹（TLS 扩展、加密套件顺序）→ 反爬手段
- **应对**：`curl_cffi`（模拟浏览器 TLS 指纹）、`tls-client`

## 三、数据存储与消息队列

### 3.1 Redis

#### 核心数据结构
| 数据结构 | 爬虫用途 | 常用命令 |
|----------|----------|----------|
| String | 缓存网页内容、计数器 | `SET`/`GET`/`INCR`/`EXPIRE` |
| Hash | 存储结构化数据（用户信息） | `HSET`/`HGET`/`HGETALL` |
| List | 任务队列（FIFO/LIFO） | `LPUSH`/`RPUSH`/`LPOP`/`BRPOP` |
| Set | URL 去重 | `SADD`/`SISMEMBER`/`SMEMBERS` |
| ZSet | 优先级队列、排行榜 | `ZADD`/`ZRANGE`/`ZRANGEBYSCORE` |
| Bitmap | 大规模去重（布隆过滤器） | `SETBIT`/`GETBIT` |
| Stream | 消息队列（Redis 5.0+） | `XADD`/`XREAD`/`XREADGROUP` |

#### 关键特性
- **持久化**：RDB（快照）vs AOF（追加日志）；混合持久化（Redis 4.0+）
- **过期策略**：惰性删除 + 定期删除
- **内存淘汰策略**：`noeviction`、`allkeys-lru`、`volatile-lru`、`allkeys-random` 等
- **Pipeline**：批量命令减少网络往返
- **Lua 脚本**：原子操作（如分布式锁实现）
- **发布/订阅**：`PUBLISH`/`SUBSCRIBE`，但不保证消息不丢失
- **Redis 集群**：主从复制、哨兵模式、Cluster 模式（哈希槽分片）

#### 爬虫中的应用
```python
import redis

r = redis.Redis(host='localhost', port=6379, db=0)

# URL 去重
def is_duplicate(url):
    return not r.sadd('urls:seen', url)

# 分布式任务队列
r.lpush('task_queue', url)
url = r.brpop('task_queue', timeout=5)

# 分布式锁
import time
def acquire_lock(name, timeout=10):
    identifier = str(time.time())
    if r.set(f'lock:{name}', identifier, nx=True, ex=timeout):
        return identifier
    return None
```

#### 布隆过滤器（Bloom Filter）
- 原理：多个哈希函数映射到位数组
- 特点：空间效率高，无假阴性，有假阳性
- Redis 实现：`RedisBloom` 模块 或 手动 Bitmap 实现

### 3.3 Kafka

#### 核心概念
| 概念 | 说明 |
|------|------|
| Broker | Kafka 服务节点 |
| Topic | 消息主题（逻辑分类） |

```
Producer → Broker(Topic/Partition) → Consumer Group → Consumer
                    ↓

```

#### 关键特性
- **高吞吐**：顺序写磁盘 + 零拷贝（`sendfile`）+ 批量发送 + 压缩

- **Consumer 偏移管理**：`__consumer_offsets` 内部 Topic 存储
- **分区策略**：Key Hash、轮询、自定义 Partitioner

#### Python 客户端
```python
# kafka-python
from kafka import KafkaProducer, KafkaConsumer

producer = KafkaProducer(bootstrap_servers=['localhost:9092'])

producer.flush()

consumer = KafkaConsumer('crawl_results', bootstrap_servers=['localhost:9092'],
                         group_id='crawler_group', auto_offset_reset='earliest')
for message in consumer:
    print(message.value)

# confluent-kafka（基于 librdkafka，性能更好）
```

```

                                        ↓

```
- **解耦**：调度器与爬虫 Worker 解耦
- **削峰填谷**：突发任务不压垮下游
- **水平扩展**：增加 Consumer 即可扩容

### 3.4 RabbitMQ

#### 核心概念
| 概念 | 说明 |
|------|------|
| Producer | 消息生产者 |
| Exchange | 消息交换机（路由规则） |
| Queue | 消息队列（存储消息） |
| Binding | Exchange 与 Queue 的绑定关系 |
| Routing Key | 消息路由键 |
| Consumer | 消息消费者 |
| VHost | 虚拟主机（逻辑隔离） |

#### Exchange 类型
| 类型 | 路由规则 |
|------|----------|
| Direct | Routing Key 精确匹配 |
| Fanout | 广播到所有绑定的 Queue |
| Topic | 通配符匹配（`*` 一个单词，`#` 零或多个单词） |
| Headers | 基于消息 Header 属性匹配 |

#### 消息流转
```
Producer → Exchange (根据类型+Routing Key) → Queue → Consumer
```

#### 关键特性
- **消息确认（ACK）**：自动确认 vs 手动确认（`basic_ack`）
- **持久化**：Exchange + Queue + Message 三者都需要设置持久化
- **死信队列（DLX）**：消息被拒绝/过期/队列满时转发到死信交换机
- **延迟消息**：`rabbitmq-delayed-message-exchange` 插件 或 TTL + DLX 实现
- **优先级队列**：`x-max-priority` 设置队列优先级范围
- **预取（QoS）**：`basic_qos(prefetch_count=N)` 限制未确认消息数

#### Python 客户端
```python
import pika

# Producer
connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='crawl_tasks', durable=True)
channel.basic_publish(
    exchange='',
    routing_key='crawl_tasks',
    body='https://example.com',
    properties=pika.BasicProperties(delivery_mode=2)  # 持久化
)

# Consumer
channel.basic_qos(prefetch_count=10)
def callback(ch, method, properties, body):
    process(body)
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(queue='crawl_tasks', on_message_callback=callback)
channel.start_consuming()
```

### 3.5 Kafka vs RabbitMQ 对比

| 特性 | Kafka | RabbitMQ |
|------|-------|----------|
| 定位 | 分布式事件流平台 | 传统消息队列 |
| 吞吐量 | 极高（百万级/秒） | 中等（万级/秒） |
| 延迟 | 毫秒级 | 微秒级 |
| 消息模型 | 拉模式（Poll） | 推模式（Push） |
| 消息保留 | 持久化，可回溯 | 消费后删除 |
| 协议 | 自定义 TCP | AMQP |
| 适用场景 | 日志收集、大数据管道、事件溯源 | 任务分发、订单处理、即时通信 |
| 爬虫场景 | 大规模爬取任务分发、日志收集 | 中小型任务队列、优先级任务 |
